# Build & configuration

Authoritative refs: `docs/projectStandards/build-configuration.md`.

## Governance file map

| File | Purpose |
|---|---|
| `go.mod` / `go.sum` | Module identity + pinned versions. Every dependency requires per-module owner approval. |
| `.golangci.yml` | Linter suite — warnings = errors. Strict from day one; reactive suppression with one-line `//nolint:LINTER // reason` justification. |
| `.editorconfig` | Cross-stack formatting; Go indentation is tab (matches gofmt). |
| `sqlc.yaml` | SQL → Go code generation config. Schema = `migrations/`; queries = `internal/infra/persistence/<feature>/*.sql`; generated code lands in `internal/infra/persistence/_sqlcgen/`. |
| `Makefile` | Canonical entry points: `make ci` (the green-bar definition), `make test`, `make lint`, `make cover`, `make bench`, `make vuln`, `make sqlc`, `make migrate-*`, `make evals`. |
| `.air.toml` | Hot-reload config for `make dev` (optional). |
| `migrations/*.sql` | goose migrations (`*.up.sql` + `*.down.sql`). Applied via `make migrate-up`. Never edit an already-applied migration; write a new one. |
| `scripts/check-coverage.sh` | Coverage gate enforcer; called from `make cover`. |
| `.claude/settings.json` | Permissions + hooks + MCP wiring. Per-user overrides go in `.claude/settings.local.json` (gitignored). |

## Module discipline

- **Pin versions explicitly** in `go.mod` — never depend on `latest`.
- **Run `go mod tidy`** before every commit; the diff is part of the change.
- **`go mod why <pkg>`** before adding a new dep — confirm it's truly needed.
- **No `replace` directives** in committed `go.mod` without a documented reason.
- **`vendor/` is not committed** — relies on the module cache + `go.sum` for verification.
- The `// indirect` lines are managed by `go mod tidy`; never hand-edit them.

## Linter strictness

`.golangci.yml` enables ~25 linters. Warnings = errors via the build (`make lint` exits
non-zero on any finding).

**Tuning philosophy: maximalist, reactive.** Nothing pre-disabled. When a rule produces a
genuine false positive, suppress at the call site (not globally) with a one-line justification:

```go
//nolint:gocyclo // generated state machine; complexity is inherent
func dispatch(state State) State {
    // …
}
```

The `nolintlint` linter enforces:
- A specific linter (`//nolint:gocyclo`, not bare `//nolint`)
- A `//` comment explaining why
- No leading space inside `//nolint`

Per-path exclusions live in `.golangci.yml` under `issues.exclude-rules`:
- `_test.go` — looser cyclomatic / length / errcheck / contextcheck (tests legitimately do
  things that would be smelly in production).
- `*.sql.go` — generated by sqlc; treat as vendor.
- `cmd/*/main.go` — exempt from "must comment exported" (main has no exported symbols).

## Build pipeline

Run, in order, after every cohesive unit:

```bash
gofumpt -w .
goimports -w -local {{ProjectName}} .
go mod tidy
go vet ./...
go build ./...
golangci-lint run
go test -race -count=1 ./...
govulncheck ./...
```

`make ci` runs the whole thing. CI must mirror this exactly — any divergence is a bug.

## Migrations (goose)

- Migrations live in `migrations/` as `NNNNNN_description.sql` (timestamp-prefixed).
- Each file has an `-- +goose Up` block and a `-- +goose Down` block.
- **Down migrations must be real** — `goose down` is gated by the protect-commands hook
  (asks for confirmation) but it must actually work in dev.
- **Never edit an applied migration.** Write a new migration that fixes the prior one.
- After writing a migration, run `make sqlc` to regenerate typed Go for any new query.
- Tests apply the full migration set against the testcontainers PostgreSQL instance — a
  failing migration fails the integration suite, which is the right place to find out.

## sqlc

- Queries live next to the feature: `internal/infra/persistence/<feature>/queries/*.sql`.
- Each `.sql` file declares queries with sqlc annotations:
  ```sql
  -- name: GetProjectByID :one
  SELECT * FROM projects WHERE id = $1 AND tenant_id = $2;

  -- name: ListProjects :many
  SELECT * FROM projects WHERE tenant_id = $1 ORDER BY created_at DESC LIMIT $2 OFFSET $3;

  -- name: CreateProject :exec
  INSERT INTO projects (id, tenant_id, name, created_at) VALUES ($1, $2, $3, $4);
  ```
- `make sqlc` regenerates `internal/infra/persistence/_sqlcgen/` typed Go.
- **The generated code is committed** (so reviewers see the diff) but excluded from the
  linter (`*.sql.go`).
- Repository wrappers in `internal/infra/persistence/<feature>/` adapt the generated typed
  methods to the domain interface declared in `internal/app/<feature>/`.

## Coverage gate

`make cover` runs the suite with `-covermode=atomic` and pipes the output through
`scripts/check-coverage.sh`. Per-package thresholds:

- `internal/domain/**` and `internal/app/**` → 80% (business logic; coverage is correlated
  with correctness here)
- everywhere else → 60% (boilerplate and adapters; coverage is more about regression safety)

Override per-run via `make cover COVER_MIN_DOMAIN=85`. A drop below the threshold fails CI.
Excluding a file from coverage requires a justification comment in the file itself.

## When you change a build file

- A new linter? Add it to `.golangci.yml`, fix the new findings, document in this rule.
- A new module? Add to `go.mod`, justify in the plan / commit, update the security audit
  (every module is fresh attack surface — `govulncheck` will tell you).
- A new make target? Add to `Makefile` with a `## description` doc comment so `make help`
  finds it.
- A new migration? Apply locally, regen sqlc, write the integration test.
